# BullseyeNotify — App-Specific Bindings (BN)

Shared engineering + workflow rules live in `CLAUDE.md` (auto-synced from `../BullseyeShared` by the SessionStart `.claude/sync-shared.sh` hook; gitignored; never edit it here). TATOO short-name: **BN**. This file holds ONLY BullseyeNotify-specific bindings.

## Platform deviation — NOT on the Java/Vaadin baseline
BullseyeNotify (npm `omni-notify-mcp`) is the one Bullseye project that does NOT follow `PLATFORM.md` — no JDK / Spring / Vaadin / Maven. It is a **Node 18+ / TypeScript** MCP notification server.
- Shared-currency is kept by the **SessionStart `.claude/sync-shared.sh` hook** (copies `CLAUDE.md` + `.claude/rules.sh` from `../BullseyeShared`), not a Maven `validate` antrun. There are no `bullseye-*` libs to `mvn install`.
- Build: `npm run build` — `tsc` (`src/` → `dist/`) then `tsc -p ui/tsconfig.json` (`ui/` → `dist/ui/`).

## Architecture
- `src/index.ts` → `dist/index.js` — stdio MCP bridge (`omni-notify-mcp` bin). Declares the `claude/channel` capability, auto-spawns the UI server, subscribes to the inbox SSE. Registered in Claude Code (user scope) as stdio `node .../BullseyeNotify/dist/index.js`, env `NOTIFY_MCP_TAG=claude-code`.
- `ui/server.ts` → `dist/ui/server.js` — HTTP/UI server on `:3737`; all channel impls; the Streamable-HTTP `/mcp` endpoint; the inbound inbox; the Telegram listener AND the always-on Slack poller (`startSlackListener`).

## Build + relaunch — server MUST be relaunched after every server change (Meni mandate 2026-06-04, verbatim: *"when you make code changes, server is relaunched!!! otherwise we will be chasing ghosts!!!"*)
A `ui/server.ts` / `src/index.ts` edit takes effect ONLY after the process restarts — edit + `npm run build` is NOT enough; the running JVM-equivalent keeps the STALE build until relaunched. After ANY such edit: `npm run build` → relaunch `:3737` — resolve the listening PID (`netstat -ano | grep ':3737' | grep LISTENING`), `taskkill //F //PID <pid>` (§9 by-PID), then relaunch detached **WITH `ENABLE_MCP=1`**: `ENABLE_MCP=1 nohup node dist/ui/server.js > .run/server.log 2>&1 &`. **`ENABLE_MCP=1` is MANDATORY** — the bridge sets it (`src/index.ts:76`); a manual relaunch that drops it makes `/mcp` return `404 mcp_disabled`, killing `wait_for_inbox`/`notify` (incident 2026-06-04). A server relaunch also invalidates the bridge's cached `/mcp` session (`Server not initialized` until a window reload), but the HTTP `/api/agent/*` endpoints + the detached worker keep working through it. The bridge only re-spawns the server on window reload (`spawnUiServerIfNeeded`, `detached`+`unref`), so a manual relaunch is required mid-session. Verify live (`curl :3737/v1/health` + the changed behavior in `.run/server.log`) before declaring done.

## Full-duplex inbound (any sender → this running session)
External senders reach the inbox via `POST /api/agent/inbox/inject`, `/api/slack/events`, the Telegram listener, the **always-on in-server Slack poller** (`startSlackListener`), or email reply links — each lands in the in-memory queue, the SSE stream, AND a `~/.notify-mcp/inbox/<ts>.<tag>.md` file-drop (Slack-origin drops carry a reply-curl hint → `POST /api/agent/slack/reply`).

**Slack channel bus (always-on, multi-VSC):** one shared Slack channel is the cross-machine bus. The poller lives INSIDE the always-on server (`startSlackListener`/`pollSlackOnce` in `ui/server.ts`, polling `conversations.history` every 2s) — NOT a separate script — so it survives as long as `:3737` runs, no agent session required (#16). Creds resolve `notify-secrets.json` (base64 `_b64` fields, decoded by `loadSecrets`/`decodeB64Fields`) → env `SLACK_BOT_TOKEN`/`SLACK_CHANNEL_ID` → `config.json` webhook; cursor in-memory with a 300s startup backfill; only `subtype==null` human messages (bot/webhook posts filtered to avoid loops). `list clients`/`help` answered in ≤2s via the webhook. `@<tag>`/`#<id> …` resolves to a connected client (`listActiveSessions` + SSE subscribers) and injects with that tag (untagged → broadcast); unknown client → error reply. Meni addresses a specific VSC by `@<tag>` or `#<id>`. The targeted VSC's drain (`.claude/notify-inbox-drain.sh`, keyed on its `<hostname>-<vsc-id>` tag) surfaces the message + reply instruction; the agent replies in-channel via `POST /api/agent/slack/reply` tagged `[@<tag>]` (#20).

**Lowest-common-denominator principle (Meni mandate 2026-06-04):** the ONLY delivery mechanism shared by every host (Claude Code CLI, Claude VSCode extension, Copilot, Cursor) is **MCP tools** — and a tool only delivers if SOMETHING calls it. The agent does NOT self-poll when idle, so "someone must CALL the poll." Design for the LCD; never assume host-specific push.

**🔑 The long-poll must live OUTSIDE the interactive session (Meni mandate 2026-06-04).** A blocking long-poll inside the agent OR inside a hook script FREEZES the user's interactive prompt — you can't talk to the agent while it's parked on a `wait`. So the responder is a SEPARATE detached process. Nothing external can *push into* an idle interactive VSCode session (Channels are CLI-only), so for the interactive session itself to drain it must self-drive periodic turns (`/loop` / `ScheduleWakeup` — instant poll per tick, never a blocking wait).

Delivery paths and where each actually works:
1. **MCP `wait_for_inbox` long-poll** — works in EVERY MCP host, but only with the server relaunched **WITH `ENABLE_MCP=1`** (else `/mcp` 404s) and a live bridge `/mcp` session (a server restart breaks it until window reload). Never block on it inside the agent/hook (freezes the prompt).
2. **`.claude/notify-inbox-drain.sh` hook (FALLBACK)** — Claude Code ONLY, turn-gated (fires at turn end; can't wake an idle session). Wired to UserPromptSubmit + **Stop ONLY** — the hook now no-ops on PreToolUse/PostToolUse because CC discards their output (#24788/#55889), so draining there DELETED messages without delivering them (incident 2026-06-04). Derives this session's `<hostname>-<vsc-id>` tag. Fallback for when the worker (path 3) is down.
3. **`notify-watch.sh` detached worker — PRIMARY responder (#20).** A STANDALONE detached process (NOT the agent, NOT a hook). Long-polls `GET /api/agent/inbox/wait?tag=<hostname>-<vsc-id>` (plain HTTP — no MCP, no bridge session, no reload) and per message: `*time*` queries answered directly (+TTS); everything else handed to a headless `claude -p` (run in a temp dir so project hooks don't recurse) that does the task and replies via `POST /api/agent/slack/reply`. Launch ONE detached instance: `nohup bash notify-watch.sh > .run/notify-watch.log 2>&1 &`. It counts as a connected client via its long-poll waiter tag (`slackClientTags` includes `inboxWaiters`), so `list clients`/`@N` resolve to it.

**No double-handling:** `ingestInboxEntry` writes a `~/.notify-mcp/inbox` file-drop ONLY when NO waiter took the message — so while the worker is parked it alone handles each message; the Stop-hook drain (path 2) only fires when the worker is down. **Slack ACK discipline (Meni mandate 2026-06-04):** the in-server poller posts a brief `ack` on receipt (tagged + untagged) so a missing final answer still shows the bus received it — NEVER echo the message back; the responder posts the actual answer; no second "caught/handing-off" line.
4. **Channels** `notifications/claude/channel` synthetic turn — ⛔ **CLI-ONLY (`claude --channels`). NOT available in the VSCode extension OR in Copilot — do NOT recommend channels for this project's runtime.** Verified against the official Channels reference: the bridge's notification shape is spec-correct, but the host never registers the listener without the CLI launch flag, so events are silently dropped here.
